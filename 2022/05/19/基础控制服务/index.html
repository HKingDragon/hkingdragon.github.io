消息队列服务

相关知识

  • 消息队列

    AMQP是一种标准化的信息中间件协议，全称为高级消息队列协议。可以让不同语言、不同系统的应用相互通信，并提供一个简单统一的模型和编程接口。可以采用各种语言和平台来实现自身的应用，当需要和其他系统通信时，只要 AMQP协议即可。

  • Rabbitmq消息服务

    RabbitMQ是一个基于AMQP协议的高级消息中间件，它主要的技术特点时可用性，安全性，集群，多协议支持，可视化的客户端，活跃的社会。

了解信息队列AMQP

信息队列AMQP服务架构

AMQP中三个重要的角色

  • Publisher：消息的发出者
  • Exchange：消息的传递者
  • Consumer：消息的接收者

不同的 Consumer 会创建不同的 Queue（消息队列），然后将对应的 Exchange 绑定到 Queue 上。在消息的传递过程中，Publisher 不会直接的把 Message 放到 Queue 中，也不管 Message 如何分发。Publisher 只管准备好消息，然后交给 Exchange，而 Exchange 做的事情 也很简单，一手从 Publisher 拿到消息，然后就把消息放入 Queue 中。

OpenStack的消息服务

OpenStack 采用 AMQP 作为它的消息传递技术。AMQP 的控制端（Broker），不管是 RabbitMQ 戒者 Qpid，它的作用是允许任何两个相同组件以松散耦合的方式进行交流。更准确地说，OpenStack 的组件均可以使用进程呼叫机制（RPC）进行彼此沟通。

镜像服务

概述

Glance 镜像服务实现发现、注册、获取虚拟机镜像和镜像元数据，镜像数据支持多种 存储系统，可以是简单文件系统、对象存储系统等。

服务架构

Glance 镜像服务是典型的 C/S 架构， Glance 架构包括 glance-CLIent、Glance 和 Glance Store。Glance 主要包括 REST API、数据 库抽象层（DAL）、域控制器（glance domain controller）和注册层（registry layer），Glance 使用集中数据库（Glance DB）在 Glance 各组件间直接共享数据。

  • 客户端（CLient）：外部用于同Glance服务的交互和操作
  • glance-api：Glance对外的REST接口
  • 数据库抽闲层（DAL）：Glance和数据库直接交互的编程接口
  • Glance域控制器：中间件实现Glance的认证、通知、策略和数据链等主要功能
  • 注册层：可选层，用于管理域控制和数据库DAL层之间的安全通信
  • Glance DB：存储镜像的元数据，根据需要可以选择不同类型的数据库，目前采用MySQL
  • Glance Store：GLance对接不同数据存储的抽象层。
  • 后端存储：实际接入的存储系统。可以接入简单文件系统、swift、Ceph和S3云存储等。当前框架选择存储在本地，目录在控制节点：/var/lib/glance/images

Glance 服务自带两个配置文件，在使用 Glance 镜像服务时需要配置 glance-api.conf 和 glance-registry.conf 两个服务模块。在添加镜像到 Glance 时，需要指定虚拟镜像的磁盘格式 （disk format）和容器格式（container format）

  • glance-api：对外接受并发布镜像、获取和存储的请求调用。
  • glance-registry：存储、处理和获取镜像元数据，内部服务只有Glance内部使用，不暴露给用户
  • glance-all：是对前两个进程的通用封装，操作方式和结构一样

镜像文件格式

  • raw：非结构化磁盘镜像格式
  • qcow2：QEMU模拟器支持的可动态扩展、写时复制的磁盘格式，是kvm虚拟机默认使用的磁盘文件格式
  • AMI/AKI/ARI：Amazon EC2最初支持的镜像格式
  • UEC tarball：Ubuntu enterprise cloud tarball 是一个gzip压缩后的tar文件，包含有AMI、AKI和ARI三种类型文件
  • VHD：微软虚拟磁盘文件的简称
  • VMDK：virtualbox使用VDI的镜像格式，OpenStack没有直接的支持，需要进行格式转换
  • VMDK：是虚拟机Vmare创建的虚拟机格式
  • OVF：开放虚拟化格式，ovf文件是一种开源的文件规范，可用于虚拟机文件的打包

容器格式

  • Bare：指定没有容器和元数据封装在镜像中，如果GLance和OpenStack的其他服务没有使用容器格式的字符串，为了安全，建议设置bare
  • ovf：ovf的容器模式
  • aki：存储在Glance中的是Amazon的内核镜像
  • ari：存储在Glance中的是Amazon的remdisk镜像
  • ami：存储在Glance中的是A马总的machine镜像
  • ova：存储在Glance中的是OVA的tar归档文件

镜像状态

  • Queued排队：镜像ID已经创建和注册，但是镜像数据还没有上传
  • Saving保存中：镜像数据还在上传中。
  • Active有效：镜像成功创建，状态有效可用
  • Deactivated不活的：镜像成功创建，镜像对非管理员用户不可用
  • Killed错误：上传镜像数据出错，目前不可读取。
  • Deleted被删除：镜像不可用，将被自动删除
  • Pending_delete等待删除：镜像不可用，等待将被自动删除

计算服务

简介

计算服务（Nova）表示于平台的工作负载的核心。OpenStackCompute 这个名称指的是一个特定的项目，该项目也被称为 Nova，OpenStack 的 其他组件依托 Nova，不 Nova 协同工作，组成了整个 OpenStack 于平台。

Nova，是 OpenStack 计算的弹性控制器。Nova 可以说是整个于平台最重要的组件，其 功能包括运行虚拟机实例，管理网络以及通过用户和项目来控制对于的访问。

架构介绍

Nova 服务包含了 7 个子组件，分别为：NovaAPI、NovaCert、NovaCompute、 NovaConductor 呾 NovaScheduler、NovaConsoleauth 以及 NovaVncproxy

  • NovaAPI

    NovaAPI这是一个HTTP服务，用于接收和处理客户端发送的HTTP请求；API是整个Nova组件的入口。它接受用户请求，将指令发送至信息队列（Queue），并由Nova服务的子服务执行相应操作。

  • NovaCert

    用于管理证书认证，提供兼容性保障，保证所有的应用程序都能在云上运行

  • NovaCompute

    NocaCompute是Nova核心子组件，通过与NovaClient进行交互，实现了虚拟机管理的功能。负责在计算节点上对虚拟机实例进行一系列操作，包括迁移、安全组策略和快照管理等功能。

  • NovaConductor

    NovaConductor是OpenStack中的一个RPC服务，主要提供对数据库的查询和权限分配操作

  • NovaScheduler

    完成Nova的核心调度，包括虚拟机硬件资源调度、节点调度等。同时，NovaScheduler决定了虚拟机创建的具体位置

  • NovaConsoleauth

    实现对Nova控制台（VNC）的认证操作

  • NovaVncproxy

    Nova的控制台组件，实现客户端与虚拟机实例的通信，在虚拟桌面领域运用较广泛

调度机制（scheduler）

主机过滤

open stack默认支持几种过滤策略，开发者也可以根据需要实现自己的过滤策略。nova.scheduler.filters包中的过滤器有以下几种

  • AllHostsFilter：不做任何过滤，直接返回所有可用的主机列表
  • AvailabilityZoneFilter：返回创建虚拟机参数指定的集群内的主机。
  • ComputeFilter：根据创建虚拟机规格属性选择主机。
  • CoreFilter：根据 CPU 数过滤主机。
  • IsolatedHostsFilter：根据“image_isolated”呾“host_isolated”标志选择主机。
  • JsonFilter：根据简单的 JSON 字符串指定的规则选择主机。
  • RamFilter：根据指定的 RAM 值选择资源足够的主机。
  • SimpleCIDRAffinityFilter：选择在同一 IP 段内的主机。
  • DifferentHostFilter：选择不一组虚拟机丌同位置的主机。
  • SameHostFilter：选择不一组虚拟机相同位置的主机。

权值计算

经过主机过滤后，需要对主机迚行权值的计算。根据策略选择相应的某一台主机。OpenStack 对权值的计算需要一个戒多个（Weight 值代价函数）的组合， 然后对每一个经过过滤的主机调用代价函数迚行计算，将得到的值不 Weight 值相乘，得到 最终的权值。OpenStack 将在权值最小的主机上创建一台虚拟机。
